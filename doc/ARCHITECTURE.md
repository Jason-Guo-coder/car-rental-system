# CloudRent 架构说明

> 适用范围：当前全栈项目。本文记录技术栈、目录结构、数据模型、服务约定、AI 引用机制、开发约束和验收标准。

## 1. 技术栈目录结构

### 1.1 技术栈

| 层 | 技术 |
|---|---|
| 前端 | Vue 3.4、Vite 5、TypeScript、Vue Router 4、Pinia、Element Plus、Tailwind CSS、ECharts 5 |
| 后端 | Spring Boot 3.0.10、JDK 17、Spring Security、JWT、MyBatis-Plus 3.5.5 |

## 2. 数据模型

### 2.1 权限模型

权限链路：

```text
sys_user
  -> sys_user_role
  -> sys_role
  -> sys_role_permission
  -> sys_permission
```

约定：

- `sys_permission.permission_type`：`0` 目录、`1` 菜单、`2` 按钮。
- 菜单权限控制侧边栏和可访问路由。
- 按钮权限控制新增、编辑、删除、分配、出租、归还、押金返还等操作。
- 前端允许固定产品侧边栏顺序，但必须使用后端菜单结果做权限过滤。

### 2.2 组织模型

- `sys_dept` 表维护部门树。
- 用户通过 `dept_id` 归属部门。
- 部门删除前必须检查子部门，不能直接删除带子节点部门。

### 2.3 车辆基础资料

核心表：

- `auto_maker`：厂商，包含 `order_letter`。
- `auto_brand`：品牌，关联厂商。
- `auto_info`：车辆，关联厂商、品牌，包含车牌、颜色、租金、押金、状态、图片、里程、应保次数、实保次数。
- `rental_type`：出租类型和折扣。

车辆状态约定：

| 后端值 | 前端值 | 含义 |
|---|---|---|
| `0` | `idle` | 未租 |
| `1` | `renting` | 已租 |
| `2` | `maintain` | 维保 |
| `3` | `selfUse` | 自用 |

### 2.4 业务模型

- `customer`：客户资料、手机号、身份证、生日、年龄、黑白名单。
- `rental_order`：订单，包含车辆、客户、出租时间、归还时间、起租/归还里程、押金、租金应付/实付。
- `violation`：违章记录，按车牌和违章时间关联业务。
- `maintain`：保养记录，记录保养车辆、费用、时间。

### 2.5 财务模型

财务统计不单独维护汇总表，当前从订单、押金、保养费等业务数据聚合：

- 收入：订单应收/实收。
- 押金：订单押金及押金状态。
- 成本：保养费用作为运营成本。
- 趋势：按出租时间和归还时间聚合日报/月报。

## 3. 服务层约定

### 3.1 Controller 约定

- 接口前缀保持 `/rental/...`。
- 列表页优先使用后端分页接口，不在前端拉全量后过滤。
- 写操作必须保留后端权限校验，前端权限只负责体验和第一层拦截。
- 旧前端已有接口优先复用，不为新前端随意增加重复接口。

### 3.2 Service 约定

- 出租下单必须同时创建订单并更新车辆状态为已租。
- 车辆归还必须同时更新订单归还信息、车辆状态和应保次数。
- 押金返还必须先检查租期内未处理违章。
- 删除部门、菜单、角色等有关联风险的数据前必须先做前置校验。
- 涉及多表状态一致性的服务方法应保持事务边界。

### 3.3 前端 API 约定

- `frontend/src/api/http.ts` 只处理 base URL、token、统一错误、权限写操作拦截和上传。
- `frontend/src/api/backend.ts` 负责后端字段与前端模型互转。
- `stores/` 负责分页、查询、列表状态和业务 action。
- `views/` 不直接拼后端字段，不绕过 store 写业务请求。

### 3.4 时间和金额约定

- 前端提交 `LocalDateTime` 必须使用后端可解析格式。
- 租车天数按完整时间差向上取整，最低 1 天。
- 归还里程必须大于起租里程。
- 金额展示统一保留业务整数口径，不在不同页面混用四舍五入和截断。

## 4. AI 引用机制

本项目允许使用 AI 辅助开发和文档维护，但必须遵守以下机制：

- AI 只能基于本地代码、数据库脚本和用户明确给出的资料判断项目事实。
- 禁止联网下载脚本、复制未知来源代码或引入未经确认的新依赖。
- AI 修改代码前必须先阅读相关文件，优先复用现有组件、store、API 封装和样式。
- AI 输出的实现必须经过本地命令验证，至少包含与改动相关的构建或编译检查。
- AI 生成的临时计划、过程文件、缓存和本地记录不得进入 Git 仓库。
- 如果 AI 无法确认业务事实，必须在文档或回复中说明假设，不得伪造接口或数据。

## 5. 开发约束

### 5.1 前端约束

- 不恢复旧 Vue2 前端结构。
- 不新增与 `FloatingPagination.vue`、`FilterBar.vue`、`CircleIconButton.vue` 重复的分页、筛选和图标按钮组件。
- 不在页面中硬编码统计数据作为真实业务结果。
- 所有写按钮必须经过权限判断。
- 所有写请求必须经过 `http.ts` 的权限拦截规则。
- 旧后端路径兼容应放在路由和路径工具中，不在每个页面散落判断。

### 5.2 后端约束

- 不破坏旧接口路径和返回结构，除非明确做版本迁移。
- 不在配置文件中提交数据库密码、OSS 密钥、邮箱授权码。
- 不绕过 Spring Security 和方法级权限。
- 不把前端展示字段直接写成后端硬编码假数据。
- 不删除表结构；数据库演示脚本可以重置数据，但必须保留结构。

### 5.3 数据库约束

- `database/car_rental_system.sql` 是当前全量演示脚本。
- 菜单顺序应服务于当前前端侧边栏产品顺序。
- 演示数据必须覆盖未租、已租、维保、自用、黑名单、白名单、未处理违章、已处理违章、待保养、已保养、日报/月报财务等场景。
- 权限数据必须能验证管理员、运营、财务、维保、客服、只读审计等角色差异。

### 5.4 Git 约束

- 远端仓库不维护 `.gitignore` 文件。
- 不提交 `node_modules`、`dist`、`target`、`.runtime`、`.codex`、IDE 配置和系统缓存。
- 每次提交前至少检查 `git status --short` 和相关构建命令。

## 6. 禁止破坏的逻辑

- 登录后菜单和按钮权限必须由后端权限决定。
- 只读审计不能触发任何写操作。
- 出租下单后车辆不能继续作为可租车辆出现。
- 归还结算不能允许归还里程小于或等于起租里程。
- 押金返还不能跳过未处理违章检查。
- 保养页面不能把保养记录误当作待办事项。
- 总览统计不能被当前分页数据污染。
- 财务日报/月报切换不能复用旧图表数据。
- 车辆颜色必须以颜色编码对应颜色展示，新增/编辑必须使用颜色选择器。
- 图片上传必须使用后端 OSS 上传接口并保存返回 URL。

## 7. 验收标准

### 7.1 本地验证

```bash
cd frontend
npm run build
```

```bash
cd backend
mvn -q compile
```

### 7.2 一键启动验证

```bash
./start-cloudrent.command
```

验收点：

- 前端打开 `http://localhost:5180/login` 或自动选择的可用端口。
- 后端监听 `8888`。
- Redis 监听 `6379` 或脚本配置端口。
- 关闭窗口或 `Ctrl+C` 后，本脚本启动的前端、后端、Redis 被清理。

### 7.3 业务验收

- 管理员账号 `admin / 041009` 可完成完整业务链路。
- 只读审计账号只能查看，不能写入。
- 车辆出租、归还、押金返还、违章处理、保养记录、财务日报/月报均能使用真实接口。
- 数据库重新导入 `database/car_rental_system.sql` 后，仍可覆盖主要演示场景。
